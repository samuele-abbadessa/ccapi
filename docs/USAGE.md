## 1. What it is and how it works

ccapi is an HTTP server that exposes **Claude Code CLI** sessions via a REST API. It lets you drive conversations with Claude from scripts, agents, or applications without using the interactive terminal.

**Mental model.** For each message, the server spawns an **ephemeral** `claude -p` process (the prompt is passed via STDIN). The process handles the request, returns the response, and exits: no process stays alive between messages.

**Conversation continuity** is guaranteed by the transcript that Claude Code saves to disk, identified by a UUID generated by the server at session creation:

- The **first** invocation of a session uses `claude -p --session-id <uuid>` â†’ creates the transcript.
- **Subsequent** invocations use `claude -p --resume <uuid>` â†’ resume the context.

The server's **SQLite** registry is an *API view* (stores user prompt + final response, to expose them via `GET`). It is not the model's memory and does not re-inject context into the prompt: conversational memory lives in the Claude transcript.

## 3. Configuration

Every option can be set with a CLI flag or an environment variable. Precedence: **CLI flag > environment variable > default**.

| Option | CLI flag | Environment variable | Default |
|---|---|---|---|
| Listen port | `--port` | `CCAPI_PORT` | `4096` |
| Bind address | `--host` | `CCAPI_HOST` | `127.0.0.1` |
| Claude binary | `--claude-bin` | `CCAPI_CLAUDE_BIN` | `claude` (from PATH) |
| SQLite DB path | `--db` | `CCAPI_DB` | `.ccapi/ccapi.db` |
| Session cwd root | `--detached-cwd [base]` | `CCAPI_DETACHED_CWD` | `(disabled)` |

The cwd root (`--detached-cwd`) is an **optional** value: with a path it enables the feature using that path as the allowed root; without a value it uses the server's cwd as the root. The root `/` is **rejected** (it would void the sandbox). If the root does not exist or is not a directory the server **will not start** (fail-fast).

Examples:

```bash
# Via environment variables
CCAPI_PORT=4097 CCAPI_DB=.ccapi/project-b.db npm start

# Via CLI flags (in production, with the compiled binary)
node dist/index.js --port 4097 --host 0.0.0.0
```

**Working directory.** All sessions of an instance work in the folder from which you started the server: that is where `claude` reads files and applies changes. To work on different projects, start **one instance per folder** on different ports (see Â§8).

## 4. Concepts

**Session.** A conversation. It has an `id` (UUID), an optional `title`, and a runtime `status`:
- `idle` â€” no message being processed;
- `busy` â€” a message is being processed **or** is queued.

`status` is volatile (not persisted): after a server restart all sessions go back to `idle`, but they remain resumable.

**Message.** One turn of the conversation. Each submission produces **two** records: the `user` message (the prompt) and the `assistant` message (the response). An assistant message has a `status`: `completed`, `failed`, or `aborted`.

**Parts.** A message's content is a `parts` array, a discriminated union on the `type` field:
- `{ "type": "text", "text": "..." }` â€” text response;
- `{ "type": "structured", "data": { ... } }` â€” JSON output conforming to a schema (see Â§6).

**Per-session FIFO queue.** Messages directed to the **same** session are **serialised** (one at a time, in arrival order); **different** sessions run in parallel. This prevents corrupting the Claude transcript (see Â§8).

## 5. API reference

Base URL in examples: `http://localhost:4096`. All requests with a body use `Content-Type: application/json`. All responses are JSON.

### Sessions

#### `POST /sessions` â€” create a session

Body (optional):
```json
{ "title": "My session", "cwd": "/optional/path" }
```

| Field | Type | Required | Description |
|---|---|---|---|
| `title` | string | no | Session title. |
| `cwd` | string | no | Session working directory (relative to `base` or absolute). Requires `--detached-cwd` to be active; ignored if omitted (see Â§6). |

Response `201`:
```json
{
  "id": "0ec54bc9-4967-45bf-931b-a4177fee13bc",
  "title": "My session",
  "status": "idle",
  "cwd": "/home/user/projects/my-project",
  "createdAt": 1780752193000,
  "updatedAt": 1780752193000
}
```
The `cwd` field contains the resolved absolute path of the session (or `null` for sessions created before the detached-cwd feature). No process is started: it only creates the record and UUID.

#### `GET /sessions` â€” list sessions

Response `200`: array of sessions (most recent first), each with the runtime `status` computed at that moment.

#### `GET /sessions/:id` â€” session detail

Response `200`: the session. `404` if it does not exist.

#### `PATCH /sessions/:id` â€” update the title

Body:
```json
{ "title": "New title" }
```
(`title` can be `null` to remove it.) Response `200`: the updated session. `404` if it does not exist.

#### `DELETE /sessions/:id` â€” delete from registry

Response `204`. Removes the session and its messages from the SQLite registry. Does **not** touch Claude's `.jsonl` transcript on disk (which remains resumable). `404` if it does not exist.

#### `POST /sessions/:id/abort` â€” abort

Terminates the `claude -p` process currently running for the session (if any) and clears the pending message queue. Response `200`:
```json
{ "id": "0ec54bc9-...", "status": "idle" }
```
Pending `POST /messages` requests for that session resolve with `409` (see Â§7). `404` if the session does not exist.

### Messages

#### `POST /sessions/:id/messages` â€” send a message (synchronous)

The request **blocks** until processing is complete. If the session is busy, the message is queued and the response arrives when its turn comes.

Body:

| Field | Type | Required | Description |
|---|---|---|---|
| `prompt` | string | yes | The prompt, passed to Claude via STDIN. Max 10 MB. |
| `model` | string | no | Alias (`opus`, `sonnet`, `haiku`) or full name. Default: Claude's default. |
| `effort` | string | no | `low` \| `medium` \| `high` \| `xhigh` \| `max`. |
| `outputFormat` | string | no | `text` (default) or `json`. |
| `jsonSchema` | object | no | JSON Schema for structured output. Valid **only** with `outputFormat: "json"`. |

Example:
```json
{ "prompt": "Explain in one line what a closure is.", "model": "haiku" }
```
Response `200`:
```json
{
  "info": {
    "id": "8a1d4a14-8edf-4941-a8eb-c259b65ce3f8",
    "sessionId": "0ec54bc9-4967-45bf-931b-a4177fee13bc",
    "role": "assistant",
    "status": "completed",
    "model": null,
    "costUsd": null,
    "usage": null,
    "error": null,
    "createdAt": 1780752193356
  },
  "parts": [
    { "type": "text", "text": "A closure is a function that captures variables from its defining scope." }
  ]
}
```

> **Note on metadata.** In *text mode* the `model`, `costUsd`, and `usage` fields are `null`: Claude does not emit metadata without JSON output. With `outputFormat: "json"`, `costUsd` and `usage` are populated (see Â§6). The `model` field may remain `null` even in JSON mode on some CLI versions.

#### `GET /sessions/:id/messages` â€” list messages

Response `200`: array of `{ info, parts }` in chronological order (user and assistant alternating). `404` if the session does not exist.

#### `GET /sessions/:id/messages/:msgId` â€” single message

Response `200`: `{ info, parts }`. `404` if the message does not exist or does not belong to that session.

### Service

#### `GET /health`

Response `200`: `{ "status": "ok" }`. Useful for readiness checks; does not require the `claude` binary.

## 6. Usage workflows

### Multi-turn conversation (context continuity)

Context is automatic: just send multiple messages to the **same** session.

```bash
SID=$(curl -s -X POST http://localhost:4096/sessions \
  -H 'Content-Type: application/json' -d '{"title":"demo"}' | jq -r .id)

# Turn 1
curl -s -X POST http://localhost:4096/sessions/$SID/messages \
  -H 'Content-Type: application/json' \
  -d '{"prompt":"Remember the number 42. Reply only OK.","model":"haiku"}' \
  | jq -r '.parts[0].text'        # â†’ OK

# Turn 2 (remembers the previous turn)
curl -s -X POST http://localhost:4096/sessions/$SID/messages \
  -H 'Content-Type: application/json' \
  -d '{"prompt":"What number did I ask you to remember?","model":"haiku"}' \
  | jq -r '.parts[0].text'        # â†’ 42
```

### Structured JSON output

Set `outputFormat: "json"` and provide a `jsonSchema`. The response will contain a `structured` part with data conforming to the schema.

```bash
curl -s -X POST http://localhost:4096/sessions/$SID/messages \
  -H 'Content-Type: application/json' \
  -d '{
    "prompt": "What is the capital of France?",
    "model": "haiku",
    "outputFormat": "json",
    "jsonSchema": {
      "type": "object",
      "properties": { "capital": { "type": "string" } },
      "required": ["capital"]
    }
  }' | jq '.parts[0]'
```
```json
{ "type": "structured", "data": { "capital": "Paris" } }
```
In this mode `info.usage` and `info.costUsd` are populated (e.g. `{ "inputTokens": 28, "outputTokens": 398 }`).

### Choosing model and effort

```bash
curl -s -X POST http://localhost:4096/sessions/$SID/messages \
  -H 'Content-Type: application/json' \
  -d '{"prompt":"Analyze this problem...","model":"opus","effort":"high"}'
```

### Aborting a long-running message

From another terminal, while a request is in progress:

```bash
curl -s -X POST http://localhost:4096/sessions/$SID/abort
```
The interrupted `POST /messages` request returns `409` with `{ "error": { "code": "aborted", ... } }`; messages queued for that session are also cancelled with `409`.

### Reading back the history

```bash
curl -s http://localhost:4096/sessions/$SID/messages \
  | jq '[.[] | {role: .info.role, status: .info.status, text: (.parts[0].text // .parts[0].data)}]'
```

### Per-session working directory

With the `--detached-cwd <base>` flag it is possible to assign each session a different working directory within the allowed root (`base`). Useful for a single ccapi instance managing multiple projects.

**Enabling:**

```bash
# Start the server with allowed root /home/user/projects
node dist/index.js --detached-cwd /home/user/projects

# or via environment variable
CCAPI_DETACHED_CWD=/home/user/projects npm start

# Without explicit path: uses the server's cwd as root
node dist/index.js --detached-cwd
```

**Creating a session with cwd:**

```bash
# cwd relative to base (/home/user/projects/my-project)
curl -s -X POST http://localhost:4096/sessions \
  -H 'Content-Type: application/json' \
  -d '{"title": "Project A", "cwd": "my-project"}' | jq .

# absolute cwd (must be inside base)
curl -s -X POST http://localhost:4096/sessions \
  -H 'Content-Type: application/json' \
  -d '{"title": "Project B", "cwd": "/home/user/projects/other-project"}' | jq .
```

The response includes the `cwd` field with the resolved absolute path. All messages for that session will run in that folder.

**Validation errors:**

| Code | Cause |
|---|---|
| `detached_cwd_disabled` | `cwd` in body but `--detached-cwd` is not active. |
| `invalid_cwd` | The path does not exist or is not a directory. |
| `cwd_outside_base` | The path (including via `..` or symlinks) is outside the allowed root. |

**Security note.** The sandbox is a guard-rail against mistakes and a reasonable boundary in a local/trusted model: it is not a strong jail. Validation happens at creation; symlinks created afterwards are not re-checked. The server will not start if the indicated root does not exist, is not a directory, or is `/` (the last case would void the sandbox).

## 7. Error handling

On error the response has the form:
```json
{ "error": { "code": "<code>", "message": "<description>" } }
```

| HTTP status | `code` | When |
|---|---|---|
| `400` | `invalid_body` | Invalid body (e.g. `jsonSchema` without `outputFormat: "json"`). |
| `400` | `prompt_too_large` | Prompt exceeds the 10 MB limit. |
| `400` | `detached_cwd_disabled` | `cwd` field in `POST /sessions` body but `--detached-cwd` is not active. |
| `400` | `invalid_cwd` | The requested `cwd` does not exist or is not a directory. |
| `400` | `cwd_outside_base` | The requested `cwd` (including via `..` or symlinks) is outside the allowed root. |
| `404` | `not_found` | Session or message does not exist. |
| `409` | `aborted` | The message was interrupted by an abort. |
| `502` | `process_error` | The `claude` process exited with an error (exit â‰  0); `message` includes stderr. |
| `500` | â€” | Unexpected internal server error. |

On `failed` (502) and `aborted` (409) branches the assistant message is still **persisted** with its `status` and the `error` field set, so the history remains queryable.

Example process error (e.g. tool requiring a permission not granted):
```json
{ "error": { "code": "process_error",
  "message": "claude exited with exit code 1: Error: ..." } }
```

## 8. Operational notes and limits

**One instance per project.** All sessions of an instance share the server's working directory. To work on multiple projects, start multiple instances on different ports (and DBs):

```bash
# terminal 1 â€” project A
cd ~/projects/A && CCAPI_PORT=4096 npm --prefix ~/ccapi start
# terminal 2 â€” project B
cd ~/projects/B && CCAPI_PORT=4097 npm --prefix ~/ccapi start
```

**Concurrency.** Messages on the **same** session are serialised (FIFO queue): sending two in parallel is safe, the second waits for the first. Different sessions run in parallel.

**Tool permissions.** In headless mode, if Claude tries to use a tool that has not been pre-approved the process fails (â†’ `502`). Configure permissions in `.claude/settings.local.json` in the folder where you start the server, for example:
```json
{ "permissions": { "allow": ["Bash(npm run *)", "Read", "Edit"] } }
```

**Prompt limit.** The prompt cannot exceed **10 MB** (CLI STDIN limit): beyond that, the request is rejected with `400` before the process is started.

**Server shutdown.** The server handles a clean shutdown on `SIGINT`/`SIGTERM` (terminates child processes, closes the DB). Stop it with `Ctrl-C`. If you start it in the background, make sure to terminate the correct `tsx`/`node` process: killing only the wrapper may leave the child process listening on the port.

**Persistence.** Sessions and messages are in SQLite (`.ccapi/ccapi.db` by default): they survive restarts. The `.ccapi/` folder is excluded from version control.

**Model metadata.** On some CLI versions, `info.model` may remain `null` in JSON mode (while `usage` and `costUsd` work correctly).

## 9. Troubleshooting

**`502 process_error: "... is already in use"`** â€” this should no longer occur: the server uses `--session-id` only at creation and `--resume` for subsequent calls. If it appears, the session transcript is in an unexpected state; create a new session.

**`502 process_error` mentioning a permission/tool** â€” Claude attempted an action not permitted in headless mode. Add the tool to `.claude/settings.local.json` (see Â§8).

**Port is in use (`EADDRINUSE`)** â€” there is already an instance (or an orphaned process) on the port. Find and terminate it, or use a different port:
```bash
ss -ltnp | grep 4096        # find the listening PID
```

**Slow responses** â€” each message actually invokes the model: latency is Claude's latency. For quick tests use `"model": "haiku"`.

**Health OK but messages fail** â€” `/health` does not require the `claude` binary. Verify that `claude` is in PATH (or `CCAPI_CLAUDE_BIN`) and works: `echo "hello" | claude -p`.
